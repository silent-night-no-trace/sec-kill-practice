# 项目阶段性优化记录

本文档用于沉淀当前 `seckill` 项目已经完成的优化动作、优化目标和下一步建议，方便在后续继续迭代时快速理解“为什么这样改”和“还差什么”。

## 1. 优化原则

- 不改变既有 REST API 契约，优先做内部实现收紧。
- 优先处理热点路径、重复逻辑和长时间运行下的稳态问题。
- 每轮优化都要求能被测试、构建或指标验证支撑，而不是只做理论层面的重构。

## 2. 阶段一：热点路径与重复逻辑收敛

### 2.1 数据库库存扣减收紧

- 目标：减少同步下单时的数据库往返和高并发下的自旋重试开销。
- 调整：将库存扣减改为基于 `available_stock > 0` 的单条原子更新，同时在 SQL 中递增 `version` 字段。
- 收益：热点路径从“读当前库存 -> 尝试更新 -> 失败后重试”收敛为一次写操作，降低数据库压力与重试成本。

### 2.2 活动访问与可购买校验复用

- 目标：避免同步购买与异步购买各自维护一套活动加载和时间窗校验逻辑。
- 调整：抽出 `SeckillEventAccessService`，统一承载 `getRequiredEvent` 与 `validatePurchasable`。
- 收益：减少重复代码，后续新增活动校验规则时只需要改一处。

### 2.3 保护层过期清理限频

- 目标：避免每次验证码、令牌、购买请求都全量扫描内存态缓存。
- 调整：过期清理改为按周期触发，而不是每次请求都执行。
- 收益：入口防护链路的固定开销下降，更适合持续流量场景。

## 3. 阶段二：运行态内存与代码复用优化

### 3.1 统一紧凑 ID 生成

- 目标：去掉散落在多个服务中的重复 UUID 生成代码。
- 调整：新增 `IdGenerator`，统一生成不带中划线的紧凑 ID。
- 收益：下单号、异步请求号、验证码 ID、访问令牌都通过统一入口生成，便于后续替换实现。

### 3.2 限流桶回收

- 目标：避免 `rateLimitStore` 中冷 key 长期滞留，导致进程长时间运行后内存无意义增长。
- 调整：在保护层周期清理窗口中同步清理空桶和过期桶。
- 收益：单机防护逻辑在长时间运行时更稳定，避免内存只增不减。

## 4. 阶段三：细粒度性能微优化

### 4.1 去掉库存更新的额外持久化上下文开销

- 目标：降低单次原子更新的附带 JPA 开销。
- 调整：库存扣减语句不再强制 `flushAutomatically/clearAutomatically`。
- 收益：热点更新路径更轻，减少无必要的持久化上下文刷新。

### 4.2 时间毫秒获取走轻量路径

- 目标：减少限流和过期清理路径中的时间对象构造。
- 调整：将毫秒时间获取从时区转换改为直接使用 `Clock#millis()`。
- 收益：保护层高频路径对象创建更少，逻辑也更直接。

## 5. 阶段四：可观测性建设

### 5.1 业务指标打点

- 已增加下单持久化结果指标：`seckill.order.persist`
- 已增加异步入队结果指标：`seckill.async.enqueue`
- 已增加保护层拒绝原因指标：`seckill.protection.reject`
- 已增加访问令牌签发成功指标：`seckill.protection.token.issue`

### 5.2 指标统一标签

- 已通过 Micrometer common tags 注入 `application` 和 `env`，避免跨环境查询时维度不统一。

### 5.3 端点开放

- 当前已开放 `health`、`info`、`metrics`、`prometheus` 端点。
- 已接入 Prometheus registry，支持直接抓取 Prometheus 格式输出。

## 6. 阶段五：压测准备与交付物

- 已补充压测手册：`docs/load-test.md`
- 已补充通用 JMeter 计划：`scripts/jmeter/seckill-generic-plan.jmx`
- 已补充 JMeter 使用说明：`scripts/jmeter/README.md`

这意味着项目当前已经具备“能观测、能压测、能复现核心路径问题”的基本能力，而不只是本地功能演示。

## 7. 当前阶段收益总结

- 热点扣减路径更短，数据库无效重试更少。
- 同步/异步购买在活动校验规则上已经收口，后续维护成本更低。
- 保护层从“可用”提升到“更适合持续运行”。
- 当前已经可以直接通过指标看出成功、售罄、异步失败和保护层拒绝原因的分布。
- 项目文档已覆盖运行、观测、压测三个维度，便于继续扩展。

## 8. 仍然值得继续推进的方向

1. 将验证码、访问令牌、限流状态迁移到 Redis 或网关层，支撑多实例部署。
2. 为活动列表补充分页与筛选能力，避免活动规模增大后继续依赖 `findAll()`。
3. 为 Prometheus 指标补 Grafana 仪表盘和告警阈值，完善可观测性闭环。
4. 切换到 MySQL + Redis + RabbitMQ 的完整生产拓扑，并补充对账与补偿策略。
